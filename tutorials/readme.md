# What is a Tutorial?
Tutorials are practical guides that address (ideally one, or carefully bounded) achievable goals by following the steps presented. Tutorials are always learning-oriented, and can act as a lesson in which the author is the teacher (which is hard! [read more at the Diátaxis page on Tutorials](https://diataxis.fr/tutorials/).

## How to make a Tutorial that can be *learned* from
Realize first that this may be an entirely new endeavour for you, and that's ok! Think about a tutorial as a mini-lesson that is:
- meaningful
- ends in success
- logical
- is complete

### Meaningful
The user of your documentation (your student!) should feel that they have achieved something by completing the tutorial. The outcome should work (this is difficult with open source software, as we have experienced! It is a living, changing thing).

### Successful
A true tutorial is such that it can be completed following the directions given. Lay out the path to successful completion with examples of verification that things are working as you go along (read: include screenshots, example outputs); remember that what is being done is new and strange, and context is king. Keep in mind that our book is not a tutorial, it is a scaffolded guide.

### Logical
It needs to make sense. This has to do with the order, what is done, and the connections that are made. The logic and order *will* change in a tutorial over time because the software changes.

### Complete
The user of your documentation must encounter all *actions*, all *concepts*, and all *tools* that they need to use with familiarity. In a tutorial, this includes writing about what to notice, how long to wait, if something is *not* shown. Sometimes the command prompt output is enough (such as if it is *supposed* to do nothing!).

## Writing
Tutorials are not verbose. If more explanation is warranted, link to the resources. "Use x because of y" and link to y.

This is not the place for alternative methods. The title should match what is done.

### Language to Use
- *we* statements
- *we will...* describe what is going to be done
- *first, x. Now, y.* Keep the order logical and unambiguous.
- *we must do this because... see explanation* use minimal explanations
- *the output should look similar to* give clear expectations
- *notice...remember...let's check...* give cues for confirming things are going right
- *you have fixed/built/created* describe, admire, congratulate the accomplishment
