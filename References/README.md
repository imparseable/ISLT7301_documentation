# The References Directory
Reference documents are those that describe things. In the case of software and other technologies, they are about just those things, and how to use them, including (but by no means limited to): operating systems, software packages, functions, programs, *nix commands.

Reference material is not meant to be read, but consulted. [Diátaxis](https://diataxis.fr/reference/) describes it as being like a map; it won't teach you how to do something, but it will outline what the software is and what it works, what it needs to work, and how to use it the right way.

## The goals of reference
Neutrality is the hallmark of technical reference; this runs counterintuitive to the human desire to describe, explain, teach, and reflect. **Examples** are a way to illustrate usage without describing, such as providing use cases and examples of commands in the Unix command line. Using examples provides context without wordiness, and helps the user understand the documentation.

The structure of reference must reflect how the thing being discussed works. This includes how it is structured, the order it works in, its logic, and the dependencies (or what it needs) to work.

## Narrative
Reference narratives are impersonal. They describe the thing, do not anthropomorphize technologies, and include only exactly what the thing at hand is, has available as commands and options, and does. The narrative contains the facts, and only those that are relevant to the thing being described.
